This week, I spent most of my time understanding the [`mlir` C APIs and various MLIR dialects](https://github.com/certik/mlir) and other related libraries. Specifically, I focused on the reasons why different compilers choose different MLIR dialects.

## Progress and Understanding

Focusing on [Clang IR (ClangIR / CIR)](https://llvm.github.io/clangir/), I found that it is a good starting point to understand the various MLIR dialects. Clang IR is a high-level MLIR dialect that sits between Clang's AST and LLVM IR. The primary reason for its creation was that existing dialects were not able to handle the complexity of the C language.

Similarly, [Flang](https://flang.llvm.org/docs/) has two different dialects, [HLFIR](https://flang.llvm.org/docs/HighLevelFIR.html) and [FIR](https://flang.llvm.org/docs/FIRLang.html). HLFIR keeps high-level Fortran semantics (array assignments, expressions), while FIR is closer to codegen and lowers toward LLVM IR. While lowering any language, any compiler has to make a lot of decisions, and having these intermediate dialects can help us to break down the decisions into smaller steps, each decision handled at a particular stage. For example consider lowering any vector addition, ie `arr = arr + 1`. Here, there can be various forms of decisions, like using a temporary to store the result, using vector operation or breaking it into a for loop. So, an intermediate dialect can help us to break down all of these decisions into sub-steps, each decision handled at a particular stage, and defining boundaries of where to make a decision. 

So, having multiple dialects can be helpful. For LFortran, alot of these decisions are already made in the ASR passes. However, for TinyC compiler, we start with an AST, and make all decisions in the MLIR backend. So, having multiple dialects can be helpful to break down the decisions into smaller steps, each decision handled at a particular stage, and defining boundaries of where to make a decision, in the TinyC compiler. Considering these cases, I have narrowed down two primary approaches to start implementing the MLIR backend.

Based on these approaches, I have made an initial implementation of the MLIR API in LFortran codebase on the following branches:

- **mlir1** — Direct lowering to LLVM  IR: [https://github.com/YashNagda17/lfortran/tree/mlir1](https://github.com/YashNagda17/lfortran/tree/mlir1)
- **mlir2** — Lowering through multiple dialects: [https://github.com/YashNagda17/lfortran/tree/mlir2](https://github.com/YashNagda17/lfortran/tree/mlir2)

Both paths reuse LFortran’s existing LLVM IR → object code flow after MLIR translation.

I also implemented debug flags (`--show-mlir`, `--show-mlir-llvm-dialect`, `--show-llvm-from-mlir`) and an environment switch (`USE_MLIR_Upstream=1`) to compare native C lowering vs upstream MLIR passes. I have used only the necessary codefiles from the [https://github.com/certik/mlir](https://github.com/certik/mlir) repository, in these implementations, to keep the codefiles smaller.
