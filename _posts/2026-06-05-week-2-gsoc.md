The ASR dialect shall act as a boundary between LFortran’s semantic ASR and the generic high-level MLIR that eventually lowers to LLVM. That boundary exists so that ASR and MLIR ecosystem can be separate. Any individual ASR node changes result in changes only ASR to ASR dialect conversion, without affecting the later MLIR lowering stages.

The approaches I tried, and the reasoning and learnings from them are as follows:

## Approach 1: Lowering ASR nodes to FIR and HLFIR.

[FIR](https://flang.llvm.org/docs/FIRLangRef.html) and [HLFIR](https://flang.llvm.org/docs/HighLevelFIR.html) are two fortran specific dialects that are used by the [Flang compiler](https://flang.llvm.org/). FIR is a low-level dialect that is used by the Flang compiler to represent the fortran language. HLFIR is a high-level dialect that is used by the Flang compiler to represent the fortran language. Since these dialects are fortran specific, my begining approach was to try to lower the ASR nodes to these dialects.

Here, I tried lowering the ASR nodes to these dialects, and then using [certik/mlir](https://github.com/certik/mlir) dialects to lower further. However, I quickly encountered the following issues:
1. When lowering ASR nodes to FIR, I observed that FIR is a relatively lower-level dialect. While FIR is designed to support a broad range of Fortran constructs, support for some high-level constructs like `do` loops may not be direct or may require prior transformation (for example, by lowering them into control flow primitives or other FIR constructs). Thus, certain Fortran features, such as `do` loops, might not be represented explicitly in FIR but rather as a sequence of more primitive operations. Once we lower into these form, we lose the ability to represent the ASR tree in a way that is easy to understand and debug. Also, our ASR has a property that it can be converted back to a valid fortran program, and this property is lost when we lower to FIR. Hence, this approach is not suitable for our use case.
2. Next, HLFIR is a high level dialect, which was developed by Flang as lowering from fortran to FIR was a very big jump, in terms of alot of decisions to make in 1 step. However, HLFIR also had few issues with LFortran ASR.
HLFIR is made for Flang's syntax of handling Fortran after parsing and analysis, but LFortran's ASR is different. ASR has its own node types, type system, symbol tables, and rules. Many ASR nodes like `ArrayPhysicalCast` do not have a direct mapping in HLFIR. So, lowering ASR to HLFIR would need many extra translation steps. This makes things complicated and makes it difficult to keep the dialect close to ASR and easy to debug or convert back. Hence, this approach is not suitable for our use case.

## Approach-2 & Approach-3: Alternate designs for ASR dialect

Here, we can see that the syntax is not very readable. We reveal a lot of internal implementation details, and numeric IDs, which are not very useful for the user. All of the imeplemtation gets buried in the internal IDs, and the users have no way to understand the code. Majority of the fields have no important information for the end user as well. Hence, we need a clear syntax, which can be readable for the user.  

Working on the above points, more cleaner ASR dialect syntax for this program would be as follows:
```mlir
$ LFORTRAN_ASR_DIALECT_DUMP=compact lfortran a.f90 --show-mlir-asr-dialect
module {
  asr.translation_unit {
    asr.program @print_01 {
      asr.variable @a : i32[3]
        type_declaration = ""
      asr.variable @i : i32
        type_declaration = ""
      asr.variable @x : i32
        type_declaration = ""
      asr.assign @a = [1, 2, 3]
      asr.do_loop @i = 1 to 10 {
        asr.assign @x = @i + 5 : i32
      }
      asr.print @x
      asr.print @a
      asr.return
    }
  }
}
```
{: .playcard}

However, both of these approaches have a few issues:
Here, the code is readable for the user, but we drop crucial ASR information, which might be required for the further lowering. ASR Fields such as kind, intent, ABI might be required for the further lowering, and these fields are not present here in the syntax. So, based on the two approaches, we face a dilemna. Our ASR has a lot of fields, like Intent, ABI etc, which might not be needed for every variable, but when we need them, they must be present at LLVM lowering for proper handling. So, this made designing trickier, as there are two challenges:

1. Identifying the important fields in current ASR Nodes will be trickier, since we need to be prepared to handle all edge cases, combinations which might pop up. For example, Intent might be irrelevant for normal programs, but when a variable is inside a function, Intent is important.
2. Having all fields for all ASR Nodes for ASR Dialect, made readability worse.

So, here there needs to be a trade-off, regarding when a particular value is different from some default state, which I found was very tricky to determine, espeically it shall depend on runtime.

So, we need to find a way to pass all of the information to the lowering, without compromising on the readability of the code. This forms the basis for my next approach

## Approach 3: Regioned approach for the ASR Dialect.

In this approach, we divide the ASR Dialect into three different regions:
1. The SymTab region, which will contain the declarations of the variables.
2. The Metadata region, which will contain the metadata of the program. Any additional information, for example, few Write nodes might need to store additional information, like IOMSG, IOSTAT, etc, can be stored here.
3. The Body region, which will contain the actual execution logic.

This approach is similar to the SymTab approach, but with the addition of the Metadata and Body regions. This helps us to keep the logic readable, and pass the extra information to the lowering, without losing any information. Fortran has a @symbol syntax, which tells an user to go to the reference of @symbol, to get additional information here (I like to think of it as citations in any paper). So, we can use this syntax to refer to the SymTab & Metadata regions, and dump all the bigger information there, while keeping only the logical flow in the body region. This helps us to keep logic readable, and pass the extra information to the lowering, without losing any information. Based on this approach, the above code is lowered in ASR dialect as follows:

This approach has several advantages: Since each region is independent, users can easily navigate to the relevant region to get the information they need. We can dump all the bigger information in the SymTab & Metadata regions, while keeping the logic readable in the Body region. If needed, we can add a compact flag in the future, which can only print the Body region, without the SymTab & Metadata regions. So, this approach has a good balance between readability and information passing.

### Approach 3: Code implementation

Code implementation is hosted here: <a href="https://github.com/YashNagda17/lfortran/tree/mlir6" target="_blank" rel="noopener noreferrer">https://github.com/YashNagda17/lfortran/tree/mlir6</a>

While implementing this approach, I followed a few basic principles:

1. Keep ASR Dialect generation close to <code class="file-ref">ASR.asdl</code>, so each ASR constructor can map to one `asr.*` operation as much as possible.
2. Keep ASR Dialect printing separate from ASR Dialect lowering. The printed dialect should help debugging, while lowering can still use the full stored information.
3. Keep declarations, metadata, and executable logic separate using `asr.symtab`, `asr.metadata`, and `asr.body`. This keeps the body readable without losing information needed later.
4. Avoid exposing internal handles in the normal output. Users should see symbols like `@x` and structured regions, not pointer-like IDs.
5. Lowering APIs are auto-generated from <code class="file-ref">ASR.asdl</code>, so we can easily add new lowering handlers for new ASR nodes.
6. Leveraging the existing [certik/mlir](https://github.com/certik/mlir) repository for high level dialects & llvm dialects.

The main pipeline is:

```text
Initial ASR -> ASR Dialect -> high-level MLIR -> LLVM Dialect -> LLVM IR
```
{: .playcard}

These flags helped verify that the new ASR dialect is not just a pretty dump, but an actual intermediate stage in the `mlir-new` backend.

Currently, high level dialects & llvm dialects are implemented from the [certik/mlir](https://github.com/certik/mlir) repository. Once the LLVM IR is implemented, we execute it using the existing lfortran pipeline for implentation, which is used by original mlir backend. Also, currently, lowering support from ASR dialect to other mlir dialects is very limited, which will be expanded in coming weeks, once ASR dialect is stable.
